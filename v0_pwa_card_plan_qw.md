# План разработки PWA "Loyalty Card System" (Дедлайн: 4 марта)

## Введение

Цель: Создать легковесное веб-приложение (PWA) для регистрации клиентов и выдачи им виртуальных скидочных карт с QR-кодом и SMS-верификацией.

---

## Сценарии использования (User Flow)

### Три возможных сценария при открытии приложения:

#### 1. Новый клиент (первый вход)
- В БД нет такого телефона
- **Действие:** Форма регистрации → SMS → Верификация → Карта

#### 2. Зарегистрированный и верифицированный клиент (повторный вход по иконке)
- В БД есть телефон с `is_verified=True`
- **Действие:** Сразу показывать карту (без SMS, без формы)

#### 3. Зарегистрированный, но НЕ верифицированный клиент (бросил на этапе SMS)
- В БД есть телефон с `is_verified=False`
- **Действие:** Форма ввода SMS-кода (код уже отправлен при регистрации) → Верификация → Карта

---

### Логика работы приложения (схема)

```
Открытие приложения
    ↓
Есть ли активная сессия (token/cookie)?
    ├── ДА → Проверить is_verified в БД
    │         ├── True → Сразу показать карту
    │         └── False → Показать страницу ввода SMS-кода
    │
    └── НЕТ → Показать форму входа (только телефон)
              ↓
              Ввёл телефон → Проверка в БД
              ├── Нет в БД → Регистрация → SMS → Верификация → Карта
              └── Есть в БД
                  ├── is_verified=True → Сразу показать карту
                  └── is_verified=False → Отправить SMS → Верификация → Карта
```

### Механизм сессий

- **После успешной верификации** — создание сессии (cookie с session_token)
- **При открытии приложения** — проверка сессии:
  - Если сессия валидна и `is_verified=True` → показывать карту
  - Если сессия невалидна или отсутствует → форма входа по телефону
- **Хранение сессии:** localStorage + httpOnly cookie (срок: 30 дней)

## Этап 1: Слой Инфраструктуры и Базы Данных

### 1.1. Инициализация проекта
* **Цель:** Создать структуру проекта и окружение для разработки.
* **Файлы:**
  - `requirements.txt` — зависимости проекта
  - `.env.example` — шаблон переменных окружения
  - `app/__init__.py`, `app/main.py`, `app/config.py`
* **Логика:**
  1. Создать виртуальное окружение `python3 -m venv venv`
  2. Установить зависимости: `fastapi`, `uvicorn`, `sqlalchemy`, `jinja2`, `pydantic`, `python-dotenv`
  3. Создать базовую структуру папок: `app/`, `static/`, `templates/`
  4. Настроить загрузку конфигурации из `.env`

### 1.2. Создание Модели Пользователя
* **Цель:** Определить таблицу `User` для хранения данных клиентов.
* **Файл:** `app/models.py`
* **Логика:**
  - `id`: Integer (Primary Key, Autoincrement)
  - `full_name`: String(100) — Имя клиента
  - `phone`: String(20) — Уникальный индекс (нормализованный формат +7...)
  - `is_verified`: Boolean (Default: False) — Статус SMS-верификации
  - `sms_code`: String(4) — Временный код верификации (nullable)
  - `sms_code_expires_at`: DateTime — Время истечения кода (nullable)
  - `created_at`: DateTime (Default: now)

### 1.3. Настройка Подключения к БД
* **Цель:** Инициализировать SQLAlchemy с SQLite.
* **Файл:** `app/database.py`
* **Логика:**
  1. Создать `engine` с чтением URL из `.env` (SQLite: `sqlite:///./loyalty.db`)
  2. Создать `SessionLocal` (session factory)
  3. Создать `Base` для моделей
  4. Реализовать функцию `get_db()` для dependency injection
  5. Вызвать `Base.metadata.create_all(engine)` при старте
  6. **Примечание:** Для SQLite добавить `connect_args={"check_same_thread": False}`

## Этап 2: Слой API (Схемы Pydantic и Роутеры)

### 2.1. Схемы Pydantic
* **Цель:** Определить контракты для валидации входящих/исходящих данных.
* **Файл:** `app/schemas.py`
* **Логика:**
  - `UserBase`: Базовый класс (full_name, phone)
  - `UserCreate`: Для регистрации (full_name, phone с regex валидацией РФ)
  - `UserVerify`: Для SMS-кода (phone, code)
  - `UserOut`: Для ответа (id, full_name, phone, is_verified, created_at)
  - `UserListOut`: Для админ-панели (список UserOut + метаданные)

### 2.2. Роутеры API
* **Цель:** Создать эндпоинты для взаимодействия с приложением.
* **Файл:** `app/api/routers.py`
* **Логика:**
  - `GET /`: Отдача страницы регистрации (`index.html`)
  - `POST /api/register`: Регистрация пользователя → возврат `UserOut`
  - `POST /api/send-sms`: Отправка SMS с кодом → статус `{"sent": true}`
  - `POST /api/verify`: Проверка SMS-кода → `{"verified": true}` или ошибка
  - `GET /api/card/{phone}`: Данные карты для отображения (только verified)
  - `GET /admin`: Админ-панель (список пользователей)
  - `GET /admin/export`: Выгрузка базы в CSV

## Этап 3: Слой Сервисной Логики

### 3.1. CRUD Сервис
* **Цель:** Реализовать функции для работы с пользователями в БД.
* **Файл:** `app/services/crud.py`
* **Функции:**
  - `get_user_by_phone(db, phone)` → `User | None`
  - `get_user_by_id(db, user_id)` → `User | None`
  - `create_user(db, full_name, phone)` → `User`
  - `update_user(db, user, update_data)` → `User`
  - `get_all_users(db, limit, offset)` → `List[User]` (для админки)
  - `count_users(db)` → `int`
  - `delete_user(db, user)` → `bool`
  - `set_sms_code(db, user, code, expires_at)` → `User`
  - `clear_sms_code(db, user)` → `User`
  - `verify_user(db, user)` → `User`

### 3.2. SMS-Сервис
* **Цель:** Интеграция с SMS-шлюзом (SMS.ru или аналог).
* **Файл:** `app/services/sms_service.py`
* **Логика:**
  1. `generate_sms_code()` → 4-значный случайный код
  2. `send_sms(phone: str, code: str)` → отправка через API шлюза
     - Чтение API-ключа из `.env`
     - HTTP-запрос к SMS-провайдеру
     - Обработка ошибок (недостаточно средств, неверный номер)
  3. `verify_sms_code(db, phone, code)` → проверка кода и времени жизни
     - Проверка `is_verified == False`
     - Проверка `sms_code_expires_at > now`
     - Сравнение хешей кода
     - Установка `is_verified = True`, очистка кода

### 3.3. Валидация Телефона
* **Цель:** Нормализация номеров телефонов к единому формату.
* **Файл:** `app/services/phone_service.py`
* **Функции:**
  - `normalize_phone(phone: str)` → `+7XXXXXXXXXX` (удаление пробелов, скобок, тире)
  - `validate_phone(phone: str)` → `bool` (проверка по regex для РФ)

## План доработок: Механизм сессий и авто-вход (Update Plan v1.0)

### Проблема

В текущей реализации после перезагрузки страницы или повторного открытия приложения пользователь попадает на страницу регистрации, даже если уже прошёл верификацию. Необходимо реализовать механизм сессий для запоминания пользователя.

---

### Этап U1: Сервис сессий (Backend)

#### U1.1. Создание сервиса сессий
* **Цель:** Реализовать генерацию, хранение и проверку сессионных токенов.
* **Файл:** `app/services/session_service.py`
* **Функции:**
  - `create_session(db, user_id)` → `session_token` (генерация токена)
  - `get_session_by_token(db, token)` → `Session | None` (проверка токена)
  - `delete_session(db, token)` → `bool` (выход из системы)
  - `cleanup_expired_sessions(db)` → `int` (удаление просроченных сессий)
* **Логика:**
  - Токен: UUID v4 или secrets.token_urlsafe(32)
  - Срок жизни: 30 дней
  - Хранение: отдельная таблица `sessions` (user_id, token, expires_at, created_at)

#### U1.2. Обновление модели User
* **Цель:** Добавить связь с сессиями.
* **Файл:** `app/models.py`
* **Изменения:**
  - Добавить модель `Session` (id, user_id, token, expires_at, created_at)
  - Добавить relationship в модель `User`

#### U1.3. Обновление API: вход по телефону
* **Цель:** Реализовать упрощённый вход для зарегистрированных пользователей.
* **Файл:** `app/api/routers.py`
* **Новые эндпоинты:**
  - `POST /api/login` → вход по телефону (без SMS для verified)
  - `POST /api/logout` → выход из системы (удаление сессии)
  - `GET /api/me` → получение текущего пользователя (проверка сессии)

#### U1.4. Middleware для проверки сессии
* **Цель:** Автоматическая проверка сессии для всех запросов.
* **Файл:** `app/middleware/auth.py`
* **Логика:**
  - Извлекать `session_token` из cookie
  - Проверять валидность сессии в БД
  - Добавлять `current_user` в `request.state`
  - Пропускать запрос, если сессия невалидна (обработка в роутерах)

### Этап U3: Три сценария работы (Implementation)

#### Сценарий 1: Новый клиент (первый вход)
```
GET / → index.html (форма регистрации)
POST /api/register → создание User (is_verified=False)
POST /api/send-sms → отправка кода
GET /verify?phone=... → verify.html (ввод кода)
POST /api/verify → проверка кода, установка is_verified=True
POST /api/login → создание сессии, возврат токена
localStorage.setItem('session_token', token)
Redirect → /card/{phone}
```

#### Сценарий 2: Зарегистрированный и верифицированный (повторный вход)
```
GET / → checkSession() в base.html
localStorage.getItem('session_token') → есть
GET /api/me → валидная сессия, is_verified=True
Redirect → /card/{phone} (без форм, без SMS)
```

#### Сценарий 3: Зарегистрированный, но НЕ верифицированный
```
GET / → checkSession() в base.html
localStorage.getItem('session_token') → нет или истёк
GET / → index.html (форма входа по телефону)
POST /api/login (только телефон) → проверка в БД
is_verified=False → POST /api/send-sms
GET /verify?phone=... → verify.html (ввод кода)
POST /api/verify → проверка кода, is_verified=True
POST /api/login → создание сессии
localStorage.setItem('session_token', token)
Redirect → /card/{phone}
```

### Этап U4: Безопасность и оптимизация

#### U4.1. HttpOnly Cookie (опционально)
* **Файл:** `app/api/routers.py`
* **Логика:**
  - Установка cookie: `response.set_cookie(key="session_token", value=token, httponly=True, secure=True, max_age=2592000)`
  - Чтение cookie в middleware вместо localStorage

#### U4.2. Rate Limiting для /api/send-sms
* **Файл:** `app/middleware/rate_limit.py`
* **Логика:**
  - Максимум 3 SMS в час на один номер
  - Блокировка по IP или phone

#### U4.3. HTTPS для продакшена
* **Файл:** `nginx.conf` или инструкция для Selectel
* **Логика:**
  - SSL-сертификат Let's Encrypt
  - Редирект HTTP → HTTPS
  - Secure cookie только для HTTPS
