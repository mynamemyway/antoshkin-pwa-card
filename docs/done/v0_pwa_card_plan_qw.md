# План разработки PWA "Loyalty Card System" (Дедлайн: 4 марта)

## Введение

Цель: Создать легковесное веб-приложение (PWA) для регистрации клиентов и выдачи им виртуальных скидочных карт с QR-кодом и SMS-верификацией.

---

## Сценарии использования (User Flow)

### Возможные сценарии при открытии приложения:

#### 1. Новый клиент (первый вход)
- В БД нет такого телефона, нет куки
- **Действие:** Форма регистрации → SMS → Верификация → Установка Cookie → Карта

#### 2. Зарегистрированный и верифицированный клиент (повторный вход по иконке)
- В БД есть телефон с `is_verified=True`, в браузере есть валидная кука
- **Действие:** Сразу показывать карту (без SMS, без формы)

#### 3. Зарегистрированный и верифицированный клиент (новое устройство / сессия истекла)
- В БД есть телефон с `is_verified=True`, в браузере нет валидной куки
- **Действие:** Форма ввода телефна, SMS, кнопка для повторного получения SMS → Верификация → Установка Cookie → Карта

#### 4. Зарегистрированный, но НЕ верифицированный клиент (бросил на этапе SMS)
- В БД есть телефон с `is_verified=False`, нет куки
- **Действие:** Форма ввода телефна, SMS-кода (код уже отправлен при регистрации), кнопка для повторного получения SMS → Верификация → Установка Cookie → Карта

---

### Логика работы приложения (схема)

```
Открытие приложения
    ↓
Есть ли активная сессия (cookie)?
    ├── ДА → Проверить is_verified в БД
    │         ├── True → Сразу показать карту (Сценарий 2)
    │         └── False → Показать страницу ввода SMS (Сценарий 4)
    │
    └── НЕТ → Показать форму входа (только телефон)
              ↓
              Ввёл телефон → Проверка в БД
              ├── Нет в БД → Регистрация → SMS → Верификация → Cookie → Карта (Сценарий 1)
              └── Есть в БД
                  ├── is_verified=True → Отправить SMS → Верификация → Cookie → Карта (Сценарий 3)
                  └── is_verified=False → Отправить SMS → Верификация → Cookie → Карта (Сценарий 4)
```

### Механизм сессий

- **После успешной верификации** — создание сессии (cookie с session_token)
- **При открытии приложения** — проверка сессии:
  - Если сессия валидна и `is_verified=True` → показывать карту
  - Если сессия невалидна или отсутствует → форма входа по телефону
- **Хранение сессии:** localStorage + httpOnly cookie (срок: 30 дней)

## Этап 1: Слой Инфраструктуры и Базы Данных

### 1.1. Инициализация проекта
* **Цель:** Создать структуру проекта и окружение для разработки.
* **Файлы:**
  - `requirements.txt` — зависимости проекта
  - `.env.example` — шаблон переменных окружения
  - `app/__init__.py`, `app/main.py`, `app/config.py`
* **Логика:**
  1. Создать виртуальное окружение `python3 -m venv venv`
  2. Установить зависимости: `fastapi`, `uvicorn`, `sqlalchemy`, `jinja2`, `pydantic`, `python-dotenv`
  3. Создать базовую структуру папок: `app/`, `static/`, `templates/`
  4. Настроить загрузку конфигурации из `.env`

### 1.2. Создание Модели Пользователя
* **Цель:** Определить таблицу `User` для хранения данных клиентов.
* **Файл:** `app/models.py`
* **Логика:**
  - `id`: Integer (Primary Key, Autoincrement)
  - `full_name`: String(100) — Имя клиента
  - `phone`: String(20) — Уникальный индекс (нормализованный формат +7...)
  - `is_verified`: Boolean (Default: False) — Статус SMS-верификации
  - `sms_code`: String(4) — Временный код верификации (nullable)
  - `sms_code_expires_at`: DateTime — Время истечения кода (nullable)
  - `created_at`: DateTime (Default: now)

### 1.3. Настройка Подключения к БД
* **Цель:** Инициализировать SQLAlchemy с SQLite.
* **Файл:** `app/database.py`
* **Логика:**
  1. Создать `engine` с чтением URL из `.env` (SQLite: `sqlite:///./loyalty.db`)
  2. Создать `SessionLocal` (session factory)
  3. Создать `Base` для моделей
  4. Реализовать функцию `get_db()` для dependency injection
  5. Вызвать `Base.metadata.create_all(engine)` при старте
  6. **Примечание:** Для SQLite добавить `connect_args={"check_same_thread": False}`

## Этап 2: Слой API (Схемы Pydantic и Роутеры)

### 2.1. Схемы Pydantic
* **Цель:** Определить контракты для валидации входящих/исходящих данных.
* **Файл:** `app/schemas.py`
* **Логика:**
  - `UserBase`: Базовый класс (full_name, phone)
  - `UserCreate`: Для регистрации (full_name, phone с regex валидацией РФ)
  - `UserVerify`: Для SMS-кода (phone, code)
  - `UserOut`: Для ответа (id, full_name, phone, is_verified, created_at)
  - `UserListOut`: Для админ-панели (список UserOut + метаданные)

### 2.2. Роутеры API
* **Цель:** Создать эндпоинты для взаимодействия с приложением.
* **Файл:** `app/api/routers.py`
* **Логика:**
  - `GET /`: Отдача страницы регистрации (`index.html`)
  - `POST /api/register`: Регистрация пользователя → возврат `UserOut`
  - `POST /api/send-sms`: Отправка SMS с кодом → статус `{"sent": true}`
  - `POST /api/verify`: Проверка SMS-кода → `{"verified": true}` или ошибка
  - `GET /api/card/{phone}`: Данные карты для отображения (только verified)
  - `GET /admin`: Админ-панель (список пользователей)
  - `GET /admin/export`: Выгрузка базы в CSV

## Этап 3: Слой Сервисной Логики

### 3.1. CRUD Сервис
* **Цель:** Реализовать функции для работы с пользователями в БД.
* **Файл:** `app/services/crud.py`
* **Функции:**
  - `get_user_by_phone(db, phone)` → `User | None`
  - `get_user_by_id(db, user_id)` → `User | None`
  - `create_user(db, full_name, phone)` → `User`
  - `update_user(db, user, update_data)` → `User`
  - `get_all_users(db, limit, offset)` → `List[User]` (для админки)
  - `count_users(db)` → `int`
  - `delete_user(db, user)` → `bool`
  - `set_sms_code(db, user, code, expires_at)` → `User`
  - `clear_sms_code(db, user)` → `User`
  - `verify_user(db, user)` → `User`

### 3.2. SMS-Сервис
* **Цель:** Интеграция с SMS-шлюзом (SMS.ru или аналог).
* **Файл:** `app/services/sms_service.py`
* **Логика:**
  1. `generate_sms_code()` → 4-значный случайный код
  2. `send_sms(phone: str, code: str)` → отправка через API шлюза
     - Чтение API-ключа из `.env`
     - HTTP-запрос к SMS-провайдеру
     - Обработка ошибок (недостаточно средств, неверный номер)
  3. `verify_sms_code(db, phone, code)` → проверка кода и времени жизни
     - Проверка `is_verified == False`
     - Проверка `sms_code_expires_at > now`
     - Сравнение хешей кода
     - Установка `is_verified = True`, очистка кода

### 3.3. Валидация Телефона
* **Цель:** Нормализация номеров телефонов к единому формату.
* **Файл:** `app/services/phone_service.py`
* **Функции:**
  - `normalize_phone(phone: str)` → `+7XXXXXXXXXX` (удаление пробелов, скобок, тире)
  - `validate_phone(phone: str)` → `bool` (проверка по regex для РФ)

## Этап 4: Слой Frontend & PWA

### 4.1. Базовый HTML-шаблон
* **Цель:** Создать главный layout с подключением Tailwind CSS.
* **Файл:** `templates/base.html`
* **Логика:**
  - Подключение Tailwind через CDN (для скорости разработки)
  - Meta-теги для мобильных устройств (viewport, theme-color)
  - Блок для PWA manifest и apple-touch-icon

### 4.2. Страница Регистрации
* **Цель:** Форма ввода имени и телефона.
* **Файл:** `templates/index.html`
* **Логика:**
  1. Форма с полями: Имя (text), Телефон (tel)
  2. Кнопка "Получить карту"
  3. Валидация на клиенте (HTML5 + JS)
  4. После отправки → переход на страницу ввода SMS-кода

### 4.3. Страница SMS-Верификации
* **Цель:** Ввод 4-значного кода из SMS.
* **Файл:** `templates/verify.html`
* **Логика:**
  1. Поле ввода кода (4 цифры, автофокус)
  2. Кнопка "Подтвердить"
  3. Таймер обратного отсчёта (5 минут)
  4. Кнопка "Отправить код повторно"
  5. После успеха → переход на страницу карты

### 4.4. Страница Карты
* **Цель:** Отображение виртуальной карты с QR-кодом.
* **Файл:** `templates/card.html`
* **Логика:**
  1. Отображение: Имя, Номер телефона
  2. Генерация QR-кода через `qrcode.js` (клиентская библиотека)
     - Данные QR: `{"user_id": <id>, "phone": "<phone>"}`
  3. Стилизация под пластиковую карту (градиент, логотип)
  4. Инструкция "Покажите QR-код кассиру"

### 4.5. PWA Manifest
* **Цель:** Настройка "устанавливаемости" приложения.
* **Файл:** `static/manifest.json`
* **Логика:**
  - `name`: "Antoshkin Loyalty Card"
  - `short_name`: "Antoshkin Card"
  - `display`: `standalone` (без интерфейса браузера)
  - `background_color`, `theme_color`: фирменные цвета
  - `icons`: 192x192, 512x512 (сгенерировать placeholder)
  - `start_url`: `/`

### 4.6. Service Worker (Опционально)
* **Цель:** Кэширование статики для оффлайн-режима.
* **Файл:** `static/sw.js`
* **Логика:**
  - Кэширование `manifest.json`, `icon.png`, CSS
  - Стратегия: Cache First для статики, Network First для API
