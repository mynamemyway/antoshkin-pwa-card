## Цель

Добавить третий метод авторизации `check_call`, при котором пользователь самостоятельно совершает звонок на указанный номер для подтверждения телефона. Метод активируется через переменную окружения `AUTH_METHOD=check_call`.

---

## Архитектурные изменения

### Изменения в конфигурации
- Расширить `AUTH_METHOD`: `sms` | `call` | `check_call`
- Добавить новые переменные окружения для SMS.ru Check Call API

### Изменения в базе данных
- Добавить поле `sms_check_id: Optional[str]` в модель `User`
- Поле хранит идентификатор проверки от SMS.ru (формат: `XXXXXX-XXX`)

### Новые компоненты
1. **Сервис check_call_service.py** — работа с SMS.ru Check Call API
2. **Webhook эндпоинт** — `POST /api/auth/webhook/sms-ru` для приёма уведомлений
3. **Эндпоинт статуса** — `GET /api/auth/check-call-status` для polling'а фронтендом

### Изменения в существующих компонентах
- **auth_dispatcher.py** — добавить логику выбора `check_call`
- **models.py** — добавить поле `sms_check_id`
- **routers.py** — интегрировать новый метод в поток авторизации
- **templates/verify.html** — UI для режима `check_call` (кнопка звонка вместо ввода кода)

## Этап 2: Слой Сервисной Логики

### 2.1. Создание сервиса CheckCallService
*   **Файл:** `app/services/check_call_service.py`
*   **Функции:**

#### `initiate_check_call(phone: str) -> Tuple[bool, str, str, str]`
*   **Назначение:** Инициализировать проверку звонком через SMS.ru API
*   **Параметры:**
    - `phone`: нормализованный номер `+7XXXXXXXXXX`
*   **Возвращает:** `(success: bool, check_id: str, call_phone: str, message: str)`
*   **Логика:**
    1. В тестовом режиме (`SMS_TEST_MODE=True`): вернуть стаб-данные
    2. В боевом режиме: POST-запрос к `https://sms.ru/callcheck/add`
    3. Парсинг JSON-ответа: извлечь `check_id`, `call_phone`
    4. Логирование ответа API для отладки
*   **DSA Focus:** 
    - Использовать `httpx.AsyncClient` для асинхронного HTTP-запроса
    - Timeout: 10 секунд для предотвращения зависаний
    - Обработка ошибок: таймаут, сетевые ошибки, парсинг JSON

#### `save_check_id_to_user(db: AsyncSession, user: User, check_id: str) -> None`
*   **Назначение:** Сохранить `check_id` в базу данных
*   **Логика:** 
    1. Присвоить `user.sms_check_id = check_id`
    2. Закоммитить транзакцию
*   **DSA Focus:** 
    - Асинхронная сессия SQLAlchemy для неблокирующего доступа к БД

#### `verify_webhook_payload(payload: dict) -> Tuple[Optional[int], str]`
*   **Назначение:** Обработать webhook от SMS.ru
*   **Параметры:** `payload` — данные от SMS.ru (`check_id`, `status`)
*   **Возвращает:** `(user_id: Optional[int], message: str)`
*   **Логика:**
    1. Проверить статус: код `401` = звонок подтверждён
    2. Найти пользователя по `check_id` в поле `sms_check_id`
    3. Если статус `401`: установить `is_verified=True`, очистить `sms_check_id`
    4. Вернуть `user_id` для обновления сессии
*   **DSA Focus:**
    - Поиск по точному совпадению: O(1) с индексом на `sms_check_id` или O(n) без индекса
    - Атомарное обновление: транзакция гарантирует консистентность

### 2.2. Обновление auth_dispatcher.py
*   **Файл:** `app/services/auth_dispatcher.py`

    - Время выбора: O(1) — простое сравнение строк

## Этап 3: Слой API (Схемы и Роутеры)

### 3.1. Обновление схем Pydantic
*   **Файл:** `app/schemas.py`

#### `CheckCallInitRequest(BaseModel)`
*   **Назначение:** Валидация запроса на инициацию check_call
*   **Поля:** `phone: str`

#### `CheckCallInitResponse(BaseModel)`
*   **Назначение:** Ответ фронтенду с номером для звонка
*   **Поля:** 
    - `call_phone: str` — номер для звонка
    - `call_phone_pretty: str` — форматированный номер
    - `expires_in: int` — время жизни запроса (300 секунд)

#### `CheckCallStatusResponse(BaseModel)`
*   **Назначение:** Ответ на запрос статуса верификации
*   **Поля:** 
    - `verified: bool` — статус верификации
    - `message: str` — текстовое описание

#### `SMSRuWebhookPayload(BaseModel)`
*   **Назначение:** Валидация webhook от SMS.ru
*   **Поля:** 
    - `check_id: str`
    - `status: str` (код статуса)

### 3.2. Обновление роутеров
*   **Файл:** `app/api/routers.py`

#### Эндпоинт: `POST /api/auth/init-check-call`
*   **Назначение:** Инициировать проверку звонком от пользователя
*   **Логика:**
    1. Найти пользователя по телефону
    2. Вызвать `initiate_check_call()` → получить `check_id`, `call_phone`
    3. Сохранить `check_id` в БД
    4. Вернуть номер фронтенду
*   **DSA Focus:**
    - Идемпотентность: повторный вызов для того же пользователя должен обновлять `check_id`, а не создавать дубликаты

#### Эндпоинт: `GET /api/auth/check-call-status`
*   **Назначение:** Polling статуса верификации (для фронтенда)
*   **Логика:**
    1. Получить текущего пользователя (по сессии или телефону)
    2. Проверить `is_verified` в БД
    3. Вернуть статус фронтенду
*   **DSA Focus:**
    - Short Polling: интервал 2-3 секунды
    - Нагрузка на БД: N пользователей × (1 запрос / 3 сек) = ~20 запросов/мин при 1 пользователях
    - Альтернатива: WebSocket/SSE, но избыточно для данного сценария

#### Эндпоинт: `POST /api/auth/webhook/sms-ru`
*   **Назначение:** Приём webhook от SMS.ru
*   **Логика:**
    1. Принять POST-запрос с данными `check_id`, `status`
    2. Валидировать payload
    3. Если статус `401`: найти пользователя, установить `is_verified=True`
    4. Очистить `sms_check_id`
    5. Вернуть HTTP 200 SMS.ru (подтверждение получения)
*   **DSA Focus:**
    - Безопасность: проверить IP-адрес отправителя (whitelist SMS.ru)
    - Идемпотентность: повторный webhook с тем же статусом не должен ломать логику
    - Время обработки: O(1) с индексом на `sms_check_id`

## Зависимости и ограничения

### Технические ограничения
1. **Webhook требует публичного URL:** 
   - Для локальной разработки использовать ngrok или тестировать на сервере
   - В `.env` добавить `WEBHOOK_BASE_URL` для формирования полного пути

2. **Безопасность webhook:**
   - Проверка IP-адреса отправителя (whitelist SMS.ru)
   - Валидация подписи (если SMS.ru предоставляет)

3. **Таймаут проверки:**
   - SMS.ru даёт 5 минут на звонок
   - Фронтенд должен показывать таймер обратного отсчёта

### DSA Focus: Выбор структур данных
| Структура | Использование | Обоснование |
|-----------|--------------|-------------|
| `dict` для payload webhook | Парсинг JSON от SMS.ru | O(1) доступ по ключам, встроенная поддержка JSON |
| `str` для `sms_check_id` | Хранение идентификатора | Формат фиксированный, сравнение строк O(m) где m — длина |
| `Optional[str]` | Поле модели | Явно указывает, что поле может быть пустым (type safety) |

### DSA Focus: Сложность операций
| Операция | Сложность | Обоснование |
|----------|-----------|-------------|
| Поиск пользователя по `sms_check_id` | O(1) с индексом, O(n) без | Хэш-индекс или B-tree для точного совпадения |
| Инициация check_call (HTTP запрос) | O(1) | Один запрос к внешнему API |
| Polling статуса | O(1) за запрос | Простой SELECT по primary key |
| Обработка webhook | O(1) | UPDATE одной записи по индексу |

## Риски и mitigation

| Риск | Вероятность | Mitigation |
|------|-------------|------------|
| Webhook не приходит | Средняя | Реализовать fallback: polling `/callcheck/status` как резерв |
| Пользователь не звонит | Высокая | Таймаут 5 минут, предложить resend или сменить метод |
| Дублирование check_id | Низкая | UNIQUE constraint на `sms_check_id` или проверка перед записью |
| Фронтенд не узнаёт о верификации | Средняя | Настроить polling с разумным интервалом (2-3 сек) |
