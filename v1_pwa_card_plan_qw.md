## План доработок: Механизм сессий и авто-вход (Update Plan v1.1)

### Проблема

В текущей реализации после перезагрузки страницы или повторного открытия приложения пользователь попадает на страницу регистрации, даже если уже прошёл верификацию. Необходимо реализовать механизм сессий для запоминания пользователя.

---

### Архитектура безопасности

**HttpOnly Cookie (основной метод):**
- ✅ Хранение `session_token` в cookie
- ✅ `httponly=True` — недоступно для JavaScript (защита от XSS)
- ✅ `secure=True` — передача только по HTTPS
- ✅ `samesite="lax"` — защита от CSRF
- ✅ Срок жизни: 30 дней

**localStorage (вспомогательный):**
- ✅ Хранение `phone` — для автозаполнения поля ввода
- ❌ **НЕ** хранить токены сессии

---

### Краткий технический чеклист перед запуском:
- Database: Session.user_id — это ForeignKey с индексацией для быстрого поиска.
- Cookie: Параметр samesite="lax" критичен для того, чтобы PWA корректно передавало куки при возврате из фонового режима.
- Middleware: Не забудь добавить его в стек FastAPI в main.py через app.add_middleware.


### Сценарии использования (User Flow) при открытии приложения:

#### 1. Новый клиент (первый вход)
- В БД нет такого телефона, нет куки
- **Действие:** Форма регистрации → SMS → Верификация → Установка Cookie → Карта

#### 2. Зарегистрированный и верифицированный клиент (повторный вход по иконке)
- В БД есть телефон с `is_verified=True`, в браузере есть валидная кука
- **Действие:** Сразу показывать карту (без SMS, без формы)

#### 3. Зарегистрированный и верифицированный клиент (новое устройство / сессия истекла)
- В БД есть телефон с `is_verified=True`, в браузере нет валидной куки
- **Действие:** Форма ввода телефна, SMS, кнопка для повторного получения SMS → Верификация → Установка Cookie → Карта

#### 4. Зарегистрированный, но НЕ верифицированный клиент (бросил на этапе SMS)
- В БД есть телефон с `is_verified=False`, нет куки
- **Действие:** Форма ввода телефна, SMS-кода (код уже отправлен при регистрации), кнопка для повторного получения SMS → Верификация → Установка Cookie → Карта

---

### Логика работы приложения (схема)

```
Открытие приложения
    ↓
Есть ли активная сессия (cookie)?
    ├── ДА → Проверить is_verified в БД
    │         ├── True → Сразу показать карту (Сценарий 2)
    │         └── False → Показать страницу ввода SMS (Сценарий 4)
    │
    └── НЕТ → Показать форму входа (только телефон)
              ↓
              Ввёл телефон → Проверка в БД
              ├── Нет в БД → Регистрация → SMS → Верификация → Cookie → Карта (Сценарий 1)
              └── Есть в БД
                  ├── is_verified=True → Отправить SMS → Верификация → Cookie → Карта (Сценарий 3)
                  └── is_verified=False → Отправить SMS → Верификация → Cookie → Карта (Сценарий 4)
```

### Этап U1: Сервис сессий (Backend)

#### U1.1. Обновление модели User и создание Session
* **Цель:** Добавить модель сессии и связь с пользователем.
* **Файл:** `app/models.py`
* **Изменения:**
  - Добавить модель `Session` (id, user_id, token, expires_at, created_at)
  - Добавить `ForeignKey("users.id")` в таблице sessions
  - Добавить `relationship("Session", back_populates="user", cascade="all, delete-orphan")` в модель `User`
  - Таблица `sessions` будет создана автоматически через `Base.metadata.create_all()`

#### U1.2. Создание сервиса сессий
* **Цель:** Реализовать генерацию, хранение и проверку сессионных токенов.
* **Файл:** `app/services/session_service.py`
* **Функции:**
  - `create_session(db, user_id)` → `session_token` (генерация токена)
  - `get_session_by_token(db, token)` → `Session | None` (проверка токена)
  - `delete_session(db, token)` → `bool` (выход из системы)
  - `cleanup_expired_sessions(db)` → `int` (удаление просроченных сессий)
* **Логика:**
  - Токен: UUID v4 или `secrets.token_urlsafe(32)`
  - Срок жизни: 30 дней
  - Хранение: отдельная таблица `sessions` (user_id, token, expires_at, created_at)

#### U1.3. Обновление API: вход по телефону
* **Цель:** Реализовать упрощённый вход для зарегистрированных пользователей.
* **Файл:** `app/api/routers.py`
* **Новые эндпоинты:**
  - `POST /api/login` → вход по телефону (триггер отправки SMS для S3 и S4)
    - **Логика:** Проверка телефона в БД → Если найден (любой статус `is_verified`) → Отправка SMS.
    - **Response:** 200 OK (Переход на верификацию, кука НЕ устанавливается здесь).
  - `POST /api/logout` → выход из системы (удаление сессии в БД + очистка cookie в браузере).
  - `GET /api/me` → получение текущего пользователя (проверка сессии из cookie через middleware).

#### U1.4. Middleware для проверки сессии
* **Цель:** Автоматическая проверка сессии для всех запросов.
* **Файл:** `app/middleware/auth.py`
* **Логика:**
  - Извлекать `session_token` из `request.cookies`
  - Проверять валидность сессии в БД
  - Добавлять `current_user` в `request.state`
  - Пропускать запрос, если сессия невалидна (обработка в роутерах)
* **Файл:** `app/main.py`
  - Добавить middleware в `main.py`
* **Файл:** `app/main.py`
  - Обновить GET /api/me в routers.py для использования request.state.current_user (вместо ручного чтения cookie)

### Этап U2: Обновление Frontend (Client)

#### U2.1. Обновление страницы регистрации (index.html)
* **Цель:** Проверка сессии при загрузке, упрощённый вход.
* **Файл:** `templates/index.html`
* **Изменения:**
  - При загрузке: вызов `/api/me` (браузер автоматически прикрепляет cookie)
  - Если сессия валидна → редирект на `/card/{phone}`
  - Если сессии нет → показать форму
  - После ввода телефона: проверка в БД
  - Новый пользователь → регистрация → SMS
  - Зарегистрированный verified → отправка SMS → верификация (Сценарий 3)
  - Зарегистрированный not verified → отправка SMS → верификация (Сценарий 4)
  - **localStorage:** только `localStorage.getItem('phone')` для автозаполнения поля

#### U2.2. Обновление страницы верификации (verify.html)
* **Цель:** Переход на карту после успешной верификации.
* **Файл:** `templates/verify.html`
* **Изменения:**
  - После успешного `/api/verify` → **Set-Cookie устанавливается сервером**
  - **НЕ сохранять токен в localStorage**
  - Сохранение телефона: `localStorage.setItem('phone', phone)` (для автозаполнения)
  - Редирект на `/card/{phone}`

#### U2.3. Обновление страницы карты (card.html)
* **Цель:** Проверка сессии, возможность выхода.
* **Файл:** `templates/card.html`
* **Изменения:**
  - При загрузке: вызов `/api/me` (cookie прикрепляется автоматически)
  - Если сессия невалидна → редирект на `/`
  - Добавить кнопку "Выйти" → `/api/logout` (очищает сессию на сервере)

#### U2.4. Базовый шаблон (base.html) — глобальная проверка сессии
* **Цель:** Централизованная проверка сессии для всех страниц.
* **Файл:** `templates/base.html`
* **Изменения:**
  - Добавить глобальную функцию `checkSession(pageType)` в `<script>`
  - Функция вызывает `/api/me` (браузер автоматически прикрепляет cookie)
  - **Не работает с токенами напрямую** — только через cookie
* **Подэтапы реализации:**
  - **U2.4.1:** `templates/base.html` — создание глобальной функции `checkSession(pageType)`
  - **U2.4.2:** `templates/index.html` — удалить локальную `checkSessionAndRedirect()`, вызвать `checkSession('index')`
  - **U2.4.3:** `templates/card.html` — удалить локальную `checkSession()`, вызвать `checkSession('card')`
  - **U2.4.4:** `templates/verify.html` — опционально (можно пропустить, верификация — временная страница)

* **Таблица редиректов:**

| Страница | Если сессия есть (200) | Если сессии нет (401) |
|----------|------------------------|----------------------|
| `/` (index.html) | Редирект на `/card/{phone}` | Показать форму |
| `/verify` | Пропустить (уже в процессе) | Пропустить (верификация) |
| `/card/{phone}` | Показать карту | Редирект на `/` |
| `/admin` | Показать админку | Редирект на `/` |

### Этап U3: Четыре сценария работы (Implementation)

#### Сценарий 1: Новый клиент (первый вход)
```
GET / → index.html (форма регистрации)
POST /api/register → создание User (is_verified=False)
POST /api/send-sms → отправка кода
GET /verify?phone=... → verify.html (ввод кода)
POST /api/verify → проверка кода, установка is_verified=True, создание сессии, Set-Cookie: session_token=<token>
Redirect → /card/{phone}
```

#### Сценарий 2: Зарегистрированный и верифицированный (повторный вход по иконке)

```
GET / → checkSession() в base.html
      → Вызов GET /api/me (браузер сам шлет куку)
      ← Ответ 200 OK (User найден в БД по куке)
Redirect → /card/{phone} (без форм и SMS)
```

#### Сценарий 3: Зарегистрированный и верифицированный (без куки)

```
GET / → checkSession() → 401 Unauthorized (нет куки)
GET / → index.html (форма входа по телефону)
POST /api/login → отправка SMS → Redirect на verify.html
POST /api/verify → создание сессии, Set-Cookie: session_token=<token>
Redirect → /card/{phone}
```

#### Сценарий 4: Зарегистрированный, но НЕ верифицированный (бросил на этапе SMS)
```
GET / → checkSession() → 401 Unauthorized
GET / → index.html (форма входа по телефону)
POST /api/login → отправка SMS → Redirect на verify.html
POST /api/verify → проверка кода, is_verified=True, создание сессии, Set-Cookie: session_token=<token>
Redirect → /card/{phone}
```

### Этап U4: Безопасность и оптимизация

#### U4.1. Cookie параметры (обязательно)
* **Файл:** `app/api/routers.py`
* **Логика:**
  - Установка cookie: `response.set_cookie(key="session_token", value=token, httponly=True, secure=True, samesite="lax", max_age=2592000)`
  - **Обязательно**, а не опционально — это основной метод хранения токена

#### U4.2. Rate Limiting для /api/send-sms
* **Файл:** `app/middleware/rate_limit.py`
* **Логика:**
  - Максимум 3 SMS в час на один номер
  - Блокировка по IP или phone

#### U4.3. HTTPS для продакшена
* **Файл:** `nginx.conf` или инструкция для Selectel
* **Логика:**
  - SSL-сертификат Let's Encrypt
  - Редирект HTTP → HTTPS
  - Secure cookie только для HTTPS
